# SQL Server Management Studio (SSMS) Setup for RAG App

Use **SQL Server** or **Azure SQL** with SQL Server Management Studio. The app stores documents, conversations, and messages here; vectors stay in Upstash.

---

## 0. Test your connection first

From the **RAG** folder run:

```bash
node scripts/test-sql-connection.js
```

- If you see **"Connection test passed"**, your `.env.local` and SQL Server are correct; the app will connect.
- If you see **"Connection FAILED"** (e.g. ETIMEOUT), fix SQL Server and/or the connection string (see section 5 below), then run the script again.

---

## 1. What you need (credentials)

| What | Description |
|------|-------------|
| **Server** | Your SQL Server address. Examples: `localhost`, `.\SQLEXPRESS`, `myserver.database.windows.net` (Azure) |
| **Authentication** | Either **SQL Server Authentication** (login + password) or **Windows Authentication** |
| **Database name** | A database for the app, e.g. `RAGDb` |
| **Login** (if SQL auth) | Username, e.g. `sa` or a user you created |
| **Password** (if SQL auth) | Password for that login |

---

## 2. Steps in SQL Server Management Studio

### A. Connect to the server

1. Open **SQL Server Management Studio**.
2. In **Connect to Server**:
   - **Server type:** Database Engine  
   - **Server name:** your server (e.g. `localhost` or `.\SQLEXPRESS`)  
   - **Authentication:** SQL Server Authentication (or Windows Authentication)  
   - **Login / Password:** your credentials  
3. Click **Connect**.

### B. Create a database (if needed)

1. Right‑click **Databases** → **New Database**.
2. **Database name:** e.g. `RAGDb`.
3. Click **OK**.

### C. Create the tables

1. In the left panel, select your database (e.g. `RAGDb`).
2. Click **New Query** (or File → Open → open the script file).
3. Open the script: **`scripts/sql-server-rag-schema.sql`** from this project.
4. Run it: press **F5** or click **Execute**.

You should see messages like:

- `Table [documents] created.`
- `Table [conversations] created.`
- `Table [messages] created.`
- `RAG schema setup complete.`

### D. Check the tables

Under your database → **Tables**, you should see:

- **documents** – uploaded files metadata  
- **conversations** – chat sessions  
- **messages** – chat messages

## 3. Connection string for the app

Put this in your **`.env.local`** (in the project root, same folder as `package.json`).

### SQL Server on your PC (e.g. localhost / SQLEXPRESS)

```env
# Use your actual server name, database, user, and password
AZURE_SQL_CONNECTION_STRING=Server=localhost;Database=RAGDb;User Id=sa;Password=YourPassword;Encrypt=false;TrustServerCertificate=true
```

If you use a **named instance** (e.g. SQLEXPRESS):

```env
AZURE_SQL_CONNECTION_STRING=Server=localhost\\SQLEXPRESS;Database=RAGDb;User Id=sa;Password=YourPassword;Encrypt=false;TrustServerCertificate=true
```

### Azure SQL Database

```env
AZURE_SQL_CONNECTION_STRING=Server=yourserver.database.windows.net;Database=RAGDb;User Id=youradmin;Password=YourPassword;Encrypt=true
```

### Windows Authentication (same machine as SQL Server)

```env
AZURE_SQL_CONNECTION_STRING=Server=localhost;Database=RAGDb;Integrated Security=true;TrustServerCertificate=true
```

Replace:

- **Server** – your server name (or `localhost\\SQLEXPRESS` for a named instance).  
- **Database** – the database you created (e.g. `RAGDb`).  
- **User Id** / **Password** – your SQL login (not used if you use `Integrated Security=true`).

## 5. Table names and purpose

| Table          | Purpose |
|----------------|--------|
| **documents**  | One row per uploaded file (title, file name, size, full text, status). Chunk embeddings are stored in Upstash, not here. |
| **conversations** | One row per chat (user, title, timestamps). |
| **messages**   | One row per user/assistant message; `sources` holds JSON for RAG source snippets. |

All IDs are strings (UUIDs) generated by the app. Column names use **snake_case** (`user_id`, `created_at`, etc.) to match the app code.

## 6. Connection troubleshooting (ETIMEOUT / "Failed to connect")

1. **Check the exact server name in SSMS**  
   When you connect in SSMS, note the **Server name** (e.g. `.\SQLEXPRESS` or `.\SQLEXPRESS01` or `(localdb)\MSSQLLocalDB`). Use that **exact** value in your connection string (see below for format).

2. **SQL Server service must be running**  
   - Press `Win + R`, type `services.msc`, Enter.  
   - Find **SQL Server (SQLEXPRESS)** or **SQL Server (MSSQLSERVER)** or **SQL Server (SQLEXPRESS01)**.  
   - Status should be **Running**. If not, right‑click → Start.

3. **Connection string format for a named instance**  
   In `.env.local`, use one of these (replace instance name and password as needed):
   - `Server=localhost\\SQLEXPRESS01;Database=RAGDb;User Id=ShubhUser;Password=YourPass;Encrypt=false;TrustServerCertificate=true`
   - Or `Server=.\\SQLEXPRESS01;...` (single backslash in the string is OK in some contexts; if it fails, try `localhost\\InstanceName`).

4. **Enable TCP/IP** (if connection still times out)  
   - Open **SQL Server Configuration Manager**.  
   - **SQL Server Network Configuration** → **Protocols for SQLEXPRESS01** (or your instance).  
   - Right‑click **TCP/IP** → **Enable**.  
   - Restart the SQL Server service (e.g. **SQL Server (SQLEXPRESS01)** in `services.msc`).

5. **Run the test script again**  
   After any change, run:
   ```bash
   node scripts/test-sql-connection.js
   ```
   Use the same connection string in `.env.local` that you use in the app.
